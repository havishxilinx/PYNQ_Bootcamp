GridMind — Student API Reference: Talking to the Referee Board

Status: current as of 2026-07-15, verified directly against gridmind-referee/src/messages.rs (the actual, tested wire protocol). Every message shape below is exactly what the referee sends/expects — not an approximation. Updated for the hint system redesign: pregame_riddle and free_hint_fragment are new; hint_response's shape changed from a text riddle field to two digit images.

⚠️ If you've seen gridmind_client.py in this repo: it is out of date. It was written for an earlier version of the protocol (before the referee moved to the claim-based comparison and before hints existed) and does not match what's described here. Use raw pynqp2p calls with the JSON shapes below until it's updated.

1. Connecting

import pynqp2p

pynqp2p.register("192.168.1.100:5000", "bootcamp2024")   # server IP:port + shared key, from the RC team
my_id = pynqp2p.get_id()                                   # your board's MAC address
print(f"My board ID: {my_id}")   # share this with the RC team so they can register you

Every message you send goes to your arena's referee board ID (given to you by the RC team when your match is set up) via pynqp2p.send(referee_id, json_string). You read your own inbox with pynqp2p.receive_all(), which returns a list of raw JSON strings (drains your queue — each message is delivered once). You never talk to the other team's board or to the Master directly — everything routes through your Arena Agent.

import json

def send_to_referee(referee_id, msg: dict):
    pynqp2p.send(referee_id, json.dumps(msg))

def poll():
    return [json.loads(raw) for raw in pynqp2p.receive_all()]

2. Stage 1 — Pre-Game Window

Two things happen here, both before game_start. You never send anything to the referee during this stage — both mechanics are receive-only from your side.

2a. Pre-game riddle (decides who goes first)

DirectionMessageMeaning
RECEIVE{"type":"pregame_riddle","riddle":"I have cities but no houses..."}Sent to both teams identically. You have about 2 minutes — feed it to your own LLM.

Judging is entirely manual. There is no wire message for submitting an answer — whichever team calls out the correct answer first tells the human operator out loud, and the operator records the winner. The team that solves it first goes first (reflected in game_start's teams order below).

2b. Free hint (shared, non-competitive)

Right after the riddle stage, both teams receive the same hint about one real object+location on the actual match grid — split into several QR-coded fragments you assemble yourself, identically to physically scanning printed cards with your camera:

DirectionMessageMeaning
RECEIVE{"type":"free_hint_fragment","index":0,"total":3,"qr_png_base64":"..."}One fragment of the shared hint. Sent once per fragment (index 0..total-1), identically to both teams. Decode each QR image yourself (e.g. via OpenCV against the base64-decoded PNG) and concatenate them in order — the assembled text is still a riddle, not a literal answer, describing an object and its rough grid quadrant.

3. Stage 2 — Match Start

Once your Arena Agent has both teams ready, you'll receive:

DirectionMessageMeaning
RECEIVE{"type":"game_start","teams":["alpha","beta"],"total_pairs":15}Match is starting. teams lists both team names in turn order (whoever's first here goes first).
RECEIVE{"type":"your_turn","flip_num":1}It's your turn — go ahead and flip.
RECEIVE{"type":"wait","active_team":"beta"}Not your turn. Keep watching card_revealed broadcasts (below) to build your own memory of the board.

3. Stage 3 — Your Turn: Flip, Detect, Compare, Report

This is the core loop. Key idea: you compare your own two detections yourself and submit a single claim — the referee validates it against its own golden answer key, it doesn't run its own vision model.

Step-by-step

  1. Choose a position, send a flip request: 
    send_to_referee(referee_id, {"type": "flip", "team": "alpha", "pos": "B3"})
  2. Wait — a human referee physically flips the real card. This takes a moment; don't assume it's flipped until you get the next message.
  3. You'll receive a broadcast (sent to both teams, always, regardless of whose turn it is): 
    {"type": "card_revealed", "pos": "B3"}
    Notice: no class label is included. You must run your own vision model on your own camera frame to identify what's there.
  4. Run your detection, log it locally — nothing sent yet: 
    cls1, conf1 = run_your_model(camera_frame)
  5. If this was your first flip this turn: choose your second position, repeat steps 1-4 for it (cls2).
  6. Compare cls1 vs cls2 yourself. Decide: do they match?
  7. Submit your combined claim in one message: 
    send_to_referee(referee_id, {
    "type": "report_result",
    "team": "alpha",
    "pos1": "B3", "pos2": "D5",
    "cls1": "dog", "cls2": "dog",
    "claim": "match"   # or "no_match"
})
  8. You'll get back exactly one of these:
    MessageWhat happened
    {"type":"match","cls":"dog","pos1":"B3","pos2":"D5","scorer":"alpha","scores":{"alpha":3,"beta":2},"remaining":6}Your claim was correct! Points awarded (streak scoring: 1st match in a row = +1, 2nd = +2 more, etc.). Your turn continues — loop back to choosing a card.
    {"type":"no_match","pos1":"B3","pos2":"D5","cls1":"dog","cls2":"cat","scores":{"alpha":2,"beta":2}}Either you claimed "match" and were wrong (-1 point — the cls1/cls2 here are the golden truth, telling you what you actually misdetected), or you claimed "no_match" (no penalty). Either way, your turn is over.
    {"type":"invalid","reason":"Position 'B3' already matched"}Sent right after a bad flip request (already-matched position, already-flipped-this-turn position, or a position not on the grid). Pick a different position.

Alternative: flip both cards in one request

Instead of two sequential flip round-trips, you can request both of your turn's positions at once. This is only valid as your turn's first action — you can't mix the two protocols mid-turn (e.g. one single flip followed by a flip_both).

send_to_referee(referee_id, {"type": "flip_both", "team": "alpha", "pos1": "B3", "pos2": "D5"})

Validation is atomic: if either position is invalid (already matched, off-grid, or the same position twice), you get a single invalid reply and neither card is revealed — nothing to undo, just fix the bad position and resend. On success you get two separate card_revealed broadcasts (one per position, same shape as the single-flip flow), then everything from "Run your detection" onward is identical to the step-by-step above.

Both protocols are fully supported side by side — pick whichever works better for your team. flip_both exists to cut the round-trip and vision-detection time that used to sit between your two flips (the referee no longer waits on your first card's detection before the second card can be physically flipped), so it may meaningfully reduce your total match time.

While it's NOT your turn

You still receive every card_revealed broadcast from the other team's flips. Run your own detection on those too and remember them — this is free intel for your own future turns. Per the game's shared-visibility rule, both teams always see every card that's ever flipped, whoever's turn it was.

4. Stage 4 — Hints (optional, only during YOUR active turn)

Ask for a hint about a specific object you're trying to locate:

send_to_referee(referee_id, {"type": "hint_request", "team": "alpha", "object": "boat"})
ResponseMeaning
{"type":"hint_response","row_digit_png_base64":"...","col_digit_png_base64":"..."}Accepted. Two small hand-drawn-style digit images (base64-encoded PNGs), one for the row number and one for the column number — read them yourself (or build a tiny classifier) rather than a string handed to you outright. Row letters map to numbers the same way A=1, B=2, etc. Costs 1 point.
{"type":"hint_rejected","reason":"object already fully resolved"}Both cards of that object have already been revealed to someone this match — check your own card_revealed memory, you may already know where it is. Still costs 1 point.
{"type":"hint_rejected","reason":"unknown object"}The object name you sent doesn't match any class on this grid — matching is exact and case-sensitive ("Dog" ≠ "dog"). A typo here silently costs a point same as any other rejection. Still costs 1 point.
(no response at all)Silently refused: either your score is ≤ 0, or you've already used both your hint attempts this match. No cost for a silently-refused request.

Cap: at most 2 hint attempts per match count against you (accepted or rejected both count against the cap; silently-refused ones don't).

5. Stage 5 — Game Over

{"type": "game_over", "winner": "alpha", "scores": {"alpha": 9, "beta": 6}}

The match is done. If your team advances (through pool round-robin, or into the Grand Final), you'll get a fresh game_start for the next match on the same connection — no need to re-register.

Optional: Genesis Simulated Robot Arm

This is purely cosmetic and completely separate from everything above. It never talks to the referee, never affects your score, and is entirely optional — skip this section if you just want to compete.

If you want your board to visually control a simulated robot arm while it plays (for fun, not function), connect directly to the Genesis simulation server — this is a separate HTTP connection, not routed through pynqp2p or the referee at all:

from pynqsim import SimulationClient

sim = SimulationClient(genesis_server_ip, 9005)   # ask the RC team for the IP
sim.join_competition(team_id="team_red")           # or "team_blue"

Whenever you send a real flip or flip_both to the referee, you can optionally also tell Genesis to move the arm to the same position, purely for visual effect:

send_to_referee(referee_id, {"type": "flip", "team": "alpha", "pos": "B3"})   # the real move
sim.flip_card(row=1, col=2)                                                    # cosmetic only

The referee's decision about whether cards match is based only on the report_result claim you send it, exactly as described above — Genesis has no say in it, and a Genesis call failing or being skipped never affects your match.

Complete Example: a Minimal Turn Loop

import pynqp2p, json, time

pynqp2p.register("192.168.1.100:5000", "bootcamp2024")
MY_TEAM = "alpha"
REFEREE_ID = "aa:bb:cc:dd:ee:ff"  # given to you by the RC team

def send(msg):
    pynqp2p.send(REFEREE_ID, json.dumps(msg))

def poll():
    return [json.loads(raw) for raw in pynqp2p.receive_all()]

board_memory = {}  # pos -> class, built from every card_revealed you see

def wait_for(*types, timeout=65):
    deadline = time.time() + timeout
    while time.time() < deadline:
        for msg in poll():
            if msg["type"] == "card_revealed":
                # runs for EVERY flip, yours or theirs
                board_memory[msg["pos"]] = run_your_model(get_camera_frame())
            if msg["type"] in types:
                return msg
        time.sleep(0.2)
    raise TimeoutError(f"no {types} within {timeout}s")

while True:
    turn_msg = wait_for("your_turn", "game_over")
    if turn_msg["type"] == "game_over":
        print("Game over:", turn_msg)
        break

    pos1 = choose_a_position(board_memory)   # your own strategy logic
    send({"type": "flip", "team": MY_TEAM, "pos": pos1})
    wait_for("card_revealed")                # already updates board_memory[pos1]
    cls1 = board_memory[pos1]

    pos2 = choose_a_position(board_memory, exclude=pos1)
    send({"type": "flip", "team": MY_TEAM, "pos": pos2})
    wait_for("card_revealed")
    cls2 = board_memory[pos2]

    claim = "match" if cls1 == cls2 else "no_match"
    send({
        "type": "report_result", "team": MY_TEAM,
        "pos1": pos1, "pos2": pos2, "cls1": cls1, "cls2": cls2, "claim": claim,
    })
    result = wait_for("match", "no_match")
    print(result)
    # loop continues: if "match", you'll get another your_turn; if "no_match",
    # you'll get "wait" until it's your turn again.
